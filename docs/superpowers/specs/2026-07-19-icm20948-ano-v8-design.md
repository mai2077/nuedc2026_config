# ICM-20948 Anonymous V8 Diagnostic Design

## Goal

Port the STM32 reference I2C and IMU interfaces to the MSPM0G3507 project, preserve the reference IMU function-table API, and transmit one Anonymous Communication Protocol V8 `ID=0x01` inertial-sensor frame every 20 ms through `RESV1_UART`.

## Scope

- Use the STM32 `UserII2Dev` and `UserICM20948` behavior as the compatibility reference.
- Do not use the Renesas `SCI_IIC2_*` declarations or the STM32 `HAL_I2C_*` implementation directly.
- Preserve `IMU_DATA_t`, `ATTITUDE_DATA_t`, `IMUInterface_t`, `UserICM20948`, `UserMPU9250`, and the device-selection/diagnostic functions.
- Run the application with `IMU_DEVICE_ICM20948`; retain the MPU9250 compatibility path in the migrated IMU module but do not select it on this hardware.
- Send only protocol frame `ID=0x01` during this validation stage. Do not send `ID=0x02`, `ID=0x03`, ASCII diagnostics, or custom `0xF1` frames on `RESV1_UART`.

## Hardware And Configuration

- ICM-20948 uses the existing `ICM_I2C` controller with PA28 SDA and PA31 SCL.
- The module is in I2C mode (`nCS` high) and uses address `0x68` (`AD0` low). The portable example API receives the shifted 8-bit address `0xD0`.
- The existing 100 kHz I2C controller configuration remains the source of truth; no SysConfig pin change is needed.
- `RESV1_UART` remains UART0 on PA10/PA11 at 115200 baud.
- `TIMER_0` remains TIMG7 periodic at 10 ms. The application emits a frame every two timer ticks.

## Architecture

### I2C BSP: `bsp/bsp_iic.h` and `bsp/bsp_iic.c`

The header keeps the STM32 reference public types and global interface:

- `IIC_Status_t` with `IIC_OK`, `IIC_ERR`, `IIC_BUSY`, and `IIC_TIMEOUT`.
- `IICInterface_t` containing `write`, `read`, `write_reg`, `read_reg`, and `delay_ms`.
- `pIICInterface_t` and `extern IICInterface_t UserII2Dev`.

The source binds those operations to MSPM0 DriverLib. Register writes use a controller TX transfer. Register reads write the register address without STOP, then issue a repeated-start RX transfer. Address arguments are 8-bit shifted values at the public boundary and are shifted back to 7-bit values for DriverLib. FIFO waits and transfer completion share one bounded timeout budget. A previous sticky controller error is not treated as the current transfer's error before a new transfer starts; errors are checked after the current transfer starts. Error and timeout exits request STOP, reset the controller transfer, flush both FIFOs, and clear controller error interrupts. Polling register writes are limited to seven data bytes plus the register address so the complete packet fits the eight-byte TX FIFO.

`delay_ms` uses the existing cycle-delay service and is used only during initialization and reset waits. No HAL, FreeRTOS, or Renesas headers are included.

### IMU compatibility layer: `bsp/bsp_imu.h` and `bsp/bsp_imu.c`

The migrated module preserves the STM32 reference data and function-table semantics:

```c
typedef struct { float x; float y; float z; } PrivateBuf_t;

typedef struct {
    PrivateBuf_t gyro;  /* rad/s */
    PrivateBuf_t accel; /* m/s^2 */
    PrivateBuf_t magn;  /* uT */
} IMU_DATA_t;

typedef struct { float roll; float pitch; float yaw; } ATTITUDE_DATA_t;

typedef struct {
    uint8_t (*Init)(void);
    uint8_t (*DeInit)(void);
    void (*UpdateZeroPoint_axis)(const IMU_DATA_t *point);
    void (*UpdateZeroPoint_attitude)(const ATTITUDE_DATA_t *attitude);
    void (*Update_9axisVal)(IMU_DATA_t *imudata);
    void (*UpdateAttitude)(IMU_DATA_t imudata, ATTITUDE_DATA_t *attitude);
} IMUInterface_t;
```

The ICM-20948 path uses the supplied register table and the STM32 reference settings:

- reset, then wake with `PWR_MGMT_1=0x01`;
- keep accelerometer and gyroscope enabled with `PWR_MGMT_2=0x00`;
- configure gyroscope `+/-2000 dps`, DLPF enabled, and sample divider `0x04`;
- configure accelerometer `+/-2 g` and sample divider `0x04`;
- enable I2C bypass and probe the internal AK09916 magnetometer;
- wait for the reset/startup delays before accepting samples.

`Update_9axisVal` keeps the reference units: accelerometer in m/s^2, gyroscope in rad/s, and magnetometer in uT. The function applies the reference low-pass filter and subtracts the configured zero point. ICM and MPU runtime states remain separate. `UserMPU9250` is retained for source compatibility, but the application never selects it and no MPU hardware claim is made.

Because the reference `Update_9axisVal` callback returns `void`, the port also exposes a checked helper, `bsp_imu_update_9axis_checked(IMU_DATA_t *imudata)`, returning `0` on a successful primary accel/gyro sample and `1` on a failed primary sample. The legacy callback calls this helper and discards the result, preserving source compatibility. A runtime AK09916 read failure marks the magnetometer unavailable but does not discard valid accel/gyro data needed by frame `0x01`. The application uses the checked helper so it can suppress a failed primary frame without publishing stale data.

The attitude update retains the reference quaternion/Mahony-style correction: gyroscope integration propagates the quaternion, accelerometer gravity error corrects roll and pitch through proportional/integral terms, and yaw is gyro-integrated unless a later magnetometer fusion design is added. Its time delta comes from a weakly isolated MSPM0 millisecond tick adapter backed by the 10 ms timer tick, replacing the reference FreeRTOS call.

### Legacy ICM API adapter: `bsp/icm20948.h` and `bsp/icm20948.c`

The existing `ICM20948_*` API remains available for code already using the bring-up driver. Its initialization and sample operations use the same I2C backend and the same centralized ICM configuration constants as `bsp_imu.c`; no second 500 dps configuration is permitted. `ICM20948_rawToMilliDps` is updated for the selected +/-2000 dps range. These wrappers are compatibility entry points, not a second sensor state machine.

### Anonymous protocol sender: `bsp/ano_protocol.h` and `bsp/ano_protocol.c`

The sender emits exactly one standard inertial frame:

```text
AB DD FE 01 0D 00 DATA[13] SC AC
```

The public sender accepts an `IMU_DATA_t` sample and writes raw bytes through `DEBUG_UART_writeByte`, which targets `RESV1_UART`.

Payload conversion is deterministic and saturating:

- `ACC_X/Y/Z`: `round(accel_m_s2 * 100)` as little-endian signed 16-bit cm/s^2.
- `GYR_X/Y/Z`: `round((gyro_rad_s * 180/pi) * 32768 / 2000)` as little-endian signed 16-bit counts.
- `SHOCK_STA`: `0` during this diagnostic stage.

The frame length is 13. `SC` sums bytes from `HEAD` through the final data byte modulo 256. `AC` adds each intermediate `SC` modulo 256. The sender does not output labels, CR/LF, or any ASCII status on `RESV1_UART`.

### Application flow: `app/main.c`

1. Call `SYSCFG_DL_init()` and stop the TB6612 outputs using the existing BSP initialization.
2. Select `IMU_DEVICE_ICM20948` and call `UserICM20948.Init` through `bsp_imu_get_interface()`.
3. Start `TIMER_0` and enable its generated interrupt. The ISR only increments the volatile 10 ms tick.
4. In the main loop, when two ticks have elapsed, call `bsp_imu_update_9axis_checked`, send one `ID=0x01` frame only when it returns success, and schedule the next period using unsigned tick subtraction.
5. On initialization or sample-transfer failure, suppress the invalid data frame and retry initialization or the next sample on the following schedule. Do not inject diagnostic ASCII into the binary stream.

Existing `DEBUG_UART_*` and `BT_UART_*` APIs remain available to other tests. Only this application's selected output path changes to binary `RESV1_UART` frames.

## Error Handling

- Invalid pointers and zero-length transfers return the existing I2C/driver error status.
- I2C NACK/error and timeout are bounded and recorded in the IMU diagnostic state; no polling loop is unbounded.
- ICM initialization returns failure if `WHO_AM_I` is not `0xEA`, a required configuration/I2C operation fails, or the reference AK09916 probe does not return `0x09`. Magnetometer availability is still reported in diagnostics; the `ID=0x01` accel/gyro frame does not contain magnetometer fields.
- A failed sample read does not send stale or fabricated data as a new valid frame.

## Testing And Verification

### Host tests before target code

- Test the Anonymous V8 frame builder with known signed values and verify all 21 bytes, little-endian fields, `LEN=13`, `SC`, and `AC`.
- Test saturation at the signed 16-bit boundaries and zero/negative conversions.
- Extend the existing ICM I2C register model to verify the `+/-2000 dps` configuration, accel/gyro big-endian parsing, and initialization failure behavior.
- Test the compatibility function table selection and the 20 ms scheduler arithmetic independently of hardware.

### MSPM0 source checks

- Run the project's SysConfig static checker and confirm generated names (`ICM_I2C_INST`, `RESV1_UART_INST`, `TIMER_0_INST`) without editing generated files.
- Run TI Arm Clang syntax/type checks for the new BSP and application sources.
- Confirm no STM32 HAL, FreeRTOS, or Renesas headers remain in the MSPM0 source tree.

### Hardware validation

1. Connect only the ICM-20948, MCU, and UART bridge; verify `WHO_AM_I=0xEA` and stable 20 ms binary frames.
2. Capture UART bytes with a logic analyzer and check `0xAB`, addresses, `ID=0x01`, `LEN=0x000D`, payload byte order, SC, and AC.
3. Keep the sensor still and confirm gyro values are near the calibrated zero point; rotate each axis and confirm the expected signed field changes.
4. Use the Anonymous upper computer to configure one `ID=0x01` inertial frame and confirm its waveform refreshes at approximately 50 Hz.

Hardware operation and upper-computer rendering remain unverified until the user flashes and connects the physical board.
