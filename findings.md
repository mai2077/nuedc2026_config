# ICM-20948 Findings

- `ICM_I2C` maps to I2C0 with PA28 SDA and PA31 SCL.
- Generated GPIO setup uses no internal pull resistors, so external I2C pull-ups remain required.
- `SYSCFG_DL_ICM_I2C_init()` currently sets only the I2C clock source and analog glitch filter. It does not initialize controller mode or configure the SCL timer period.
- `RESV1_UART` maps to UART0 on PA10 TX and PA11 RX at 115200 baud.
- Existing `DEBUG_UART_*` functions correctly use `RESV1_UART_INST`.
- TI SDK's MSPM0G3507 polling example enables `basicEnableController` in SysConfig. Its generated init resets the controller transfer state, configures the bus timer/FIFO thresholds and clock stretching, then enables the controller.
- The corresponding SysConfig fields are `basicEnableController = true` and `basicControllerBusSpeed`. Starting at 100 kHz is the conservative bring-up choice; 400 kHz can be enabled after hardware validation.
- The official polling example accounts for MSPM0 erratum I2C_ERR_13 by delaying three I2C functional-clock cycles after starting a controller transfer.
- ICM-20948 register access is valid 11-100 ms after power-up. The 7-bit I2C address is `0x68` when AD0 is low and `0x69` when AD0 is high.
- `WHO_AM_I` is Bank 0 register `0x00` and must return `0xEA`.
- The chip powers up in sleep mode. Bank 0 `PWR_MGMT_1` register `0x06` must clear SLEEP and use `CLKSEL=1` for full gyroscope performance.
- Bank selection uses register `0x7F`, with bank number in bits 5:4.
- Gyroscope XYZ output is six big-endian bytes starting at Bank 0 register `0x33`.
- Bank 2 `GYRO_CONFIG_1` register `0x01` selects full scale and DLPF. Recommended bring-up setting is +/-500 dps with the 51.2 Hz DLPF (`GYRO_DLPFCFG=3`, `GYRO_FS_SEL=1`, `GYRO_FCHOICE=1`), giving 65.5 LSB/(deg/s).
- Gyroscope startup from full-chip sleep is specified as 35 ms, so initialization must wait after wake-up before trusting samples.
- Datasheet register reads require a write of the register address followed by a repeated START and read transfer. MSPM0 DriverLib supports this through `DL_I2C_startControllerTransferAdvanced()` with STOP disabled for the address phase.
- The minimal driver should use bounded polling/timeouts and report NACK/timeout errors instead of hanging forever when the module is absent.
- Hardware wiring confirmed by the user: `nCS` is high (I2C mode), `AD0` is low (fixed 7-bit address `0x68`), and `FSYNC` is low (unused input grounded).
- The user approved changing I2C controller settings as long as PA28 SDA and PA31 SCL remain unchanged.
- The supplied `bsp_iic.h/.c` expose a portable interface using 8-bit shifted I2C addresses and a `UserII2Dev` function table. Its hardware port functions are weak stubs that return `BSP_IIC_STATUS_NOT_CONFIGURED`; they are not a working TI implementation by themselves.
- The supplied `bsp_imu.h/.c` expose `UserICM20948` and `UserMPU9250` function tables with `Init`, `DeInit`, zero-point, 9-axis update, and attitude update operations.
- The supplied IMU implementation depends on missing `icm_20948_reg.h` and `mpu9250_reg.h` files. Neither header is present in the supplied example directory.
- No UART, serial, packet, checksum, frame header/trailer, or binary-output implementation exists in any of the four supplied files. The requested binary frame therefore cannot be reproduced byte-for-byte from the supplied material alone.
- The example defaults to `IMU_DEVICE_MPU6500`, configures both ICM-20948 and MPU9250 paths, and requires `math` functions. The current project only needs ICM-20948, so compatibility scope must be explicit before importing unrelated MPU code.
- The example ICM path configures gyro full scale as 2000 dps and returns gyro values in rad/s; the current minimal driver configures 500 dps and returns integer milli-dps. Preserving both APIs requires one authoritative hardware configuration plus tested conversion semantics.
- The second file set supplies `icm_20948_reg.h` and `mpu9250_reg.h`, so the register-table dependency is now available.
- `bsp_iic(2).c` is an STM32 HAL implementation (`HAL_I2C_*`, `I2C_HandleTypeDef`, `hi2c1`), while `bsp_iic(2).h` includes Renesas `hal_data.h` and declares `SCI_IIC2_*` plus `i2c_master_callback_args_t`. The pair cannot compile together unchanged on STM32, Renesas, or MSPM0.
- `bsp_iic(2).c` implements only the five `UserII2Dev` function-table operations. None of the `SCI_IIC2_*` functions declared in `bsp_iic(2).h` are implemented there.
- The second IMU files still contain no UART transmit call, byte-packing routine, frame header, payload length, checksum, or upper-computer protocol. Searching for UART/serial/frame/packet/send/printf/write/transmit/checksum and common frame markers found no binary-output implementation.
- `bsp_imu(2).c` retains both ICM-20948 and MPU9250 paths and a function-table API, but directly depends on `UserII2Dev`, a platform tick, and floating-point math. A TI port must bind these explicitly rather than copying HAL/FSP includes.
- The user selected the STM32 implementation as the compatibility source and explicitly rejected using the Renesas platform declarations as a reference.
- Anonymous Communication Protocol V8.10 defines frames as `HEAD,S_ADDR,D_ADDR,ID,LEN_L,LEN_H,DATA,SC,AC`, with `HEAD=0xAB` and little-endian encoding for all multi-byte fields.
- `SC` is the 8-bit sum of every byte from `HEAD` through the end of `DATA`; `AC` accumulates the running `SC` after each of those bytes.
- Protocol frame `ID=0x01` is the standard inertial sensor frame. Its 13-byte payload is six little-endian `int16_t` values (`ACC_X/Y/Z`, `GYR_X/Y/Z`) followed by one `uint8_t` shock status.
- In frame `0x01`, acceleration uses cm/s^2 and gyroscope scaling is `32768 == 2000 deg/s`. This aligns with the STM32 example's ICM gyro configuration at +/-2000 dps.
- Protocol-defined source address `0xDD` represents an Anonymous IMU and destination `0xFE` represents the upper computer. These are appropriate defaults for the ICM diagnostic unless the user selects a custom device address.
- Existing `TIMER_0` is a 10 ms periodic TIMG7 source, so a 20 ms diagnostic period should use two timer ticks without changing SysConfig.
- `RESV1_UART` remains UART0 on PA10/PA11 at 115200 baud. A 21-byte ID 0x01 frame every 20 ms uses about 10.5 kbit/s including UART framing, well within the configured baud rate.
- The STM32 IMU example derives attitude delta time from FreeRTOS `xTaskGetTickCount()`. The bare-metal MSPM0 port needs an explicit millisecond tick adapter based on `TIMER_0`.
- The STM32 ICM initialization ignores most I2C return codes and treats a missing AK09916 magnetometer as total initialization failure. The polished port should preserve public return semantics while recording exact diagnostic stage and avoiding uninitialized sensor data.
- MSPM0G3507 I2C controller TX FIFO depth is eight bytes. The polling register-write adapter therefore exposes a seven-data-byte maximum after the one-byte register address; larger writes return `IIC_ERR` instead of claiming unsupported behavior.
- A failed no-STOP register-address phase can leave the controller/bus state wedged. Every active-transfer error and timeout now uses a shared transaction budget and performs STOP/reset/FIFO/interrupt recovery before retry.
- Anonymous frame `0x01` contains only accelerometer and gyroscope values, so a runtime AK09916 failure must not invalidate those six fields. The port disables further magnetometer reads after such a failure while retaining the valid primary sample.
